This post provides a simple step-by-step guide on how to perform **Post-Processed Kinematic (PPK)** for survey purposes.

In many field situations, **Real-Time Kinematic (RTK)** positioning is not always available. RTK requires stable internet infrastructure, additional hardware, and sometimes paid correction services. This is not always practical for agricultural fields or remote areas.

Fortunately, **PPK provides a reliable alternative**. With PPK, you only need **one GNSS receiver (as a rover)** in the field. The correction data can later be downloaded from publicly available **CORS (Continuously Operating Reference Stations)**.

This tutorial explains how to perform PPK using:

- **Emlid Reach RS3**
- **Emlid Flow** (mobile app)
- **Emlid Studio** (desktop software)

The software versions used in this tutorial:

- Emlid Studio v1.9 (Windows 11 64-bit)
- Emlid Flow v14.8 (iPhone)

Other versions may look slightly different but the workflow is the same.

---


## Terminology

For people new to GNSS, here are several key terms:

- **GPS**: The satellite navigation system operated by the United States. It is only **one** of several global navigation systems.
- **GNSS**: Global Navigation Satellite System. This includes multiple satellite systems such as: GPS (USA), GLONASS (Russia), Galileo (EU) and BeiDou (China). Modern receivers like the Reach RS3 use **all systems together** for higher accuracy.
- **RTK (Real-Time Kinematic)**: A positioning technique that receives correction data **in real time** from a nearby base station.
- **PPK (Post-Processed Kinematic)**: A positioning technique where the correction data is applied **after the survey**, using recorded GNSS logs.
- **Rover**: The GNSS receiver that you carry in the field to collect survey points.
- **Base station**: A fixed GNSS receiver with known coordinates. For PPK we typically use publicly available **CORS stations**.
- **UTC Time**: A global time standard used by GNSS systems. All satellite data is recorded in **UTC time**, not local time.

## Getting Start
**Step 0**. Before starting the survey, find a **nearby CORS station** using the [NGS Web Map](https://www.arcgis.com/apps/webappviewer/index.html?id=190385f9aadb4cf1b0dd8759893032db&extent=-30657930.8234%2C-3999003.0107%2C3507586.3314%2C12379311.914%2C102100&showLayers=17bf0a87f3a-layer-16%3B17bf0a87f38-layer-14%3B17bf0a87f36-layer-13). The distance between your rover and the base station strongly affects the accuracy.

| Distance | Quality |
|--------|--------|
| 0–10 km | Best |
| 10–20 km | Very good |
| 20–40 km | Acceptable |
| 40–60 km | Not recommended |
| >60 km | Usually unreliable |

If multiple stations are available, always choose the **closest one**.

**Step 1**. Connect your phone to the GNSS receiver on *Emlid Flow*, make sure the GNSS receiver is logging raw RINEX data, and create survey project in ⌈Survey⌋. You do not need RTK correction during the survey.

You might also find this helpful: [How to log data for PPK](https://docs.emlid.com/reachrs3/ppk-quickstart/how-to-record-and-download-logs/ppk-logging/)
{: .notice}

**Step 2**. Tap the ⚙️ gear icon to configure survey settings
  - `FIX ONLY`: Off
  - `COLLECT MODE`: Averaging, 5s  

<div class="notice--info">
<p><strong style="color:#8a3b00;">Why averaging is important:</strong></p>
<p>CORS stations usually record GNSS data every <strong>5 seconds</strong>. If you collect survey points shorter than the base station sampling interval, the PPK software may not have enough data to compute corrections.</p>
<p>General rule: Averaging duration ≥ base station sampling interval</p>
<p>Example: If the base station sample rate is **8 seconds**, set averaging to <strong>≥8 seconds</strong></p>
</div>

**Step 3**. Export your survey project from *Emlid Flow* on your phone as a **CSV file**. This file contains the raw coordinates recorded in the field. Later, PPK will correct these coordinates.

**Step 4**. Download the raw GNSS log from your receiver (rover) on your phone. Typical files include:
- xxxx.26O (observation file)
- xxxx.26P (navigation file)
- xxxx.26B (backup file)

The files we need are:
- .26O
- .26P

Explanation:

| File | Purpose |
|----|----|
| .26O | Rover observation data |
| .26P | Satellite navigation data |

The number **26** represents the year **2026**. For example: 21O -> 2021; 28O -> 2028.

**Step 5**. Download the logging RINEX data that cover the survey hours from your selected base station in CORS Network.
    - Option 1: Go to the detail page of your selected base station, and click ⌈Quick Data Download⌋.
    - Option 2: Download from [NCN Data and Product](https://geodesy.noaa.gov/CORS/data.shtml).

Details of how to download correct files...

**Step 6**. Launch *Emlid Studio*, and import your Rover logging RINEX data, Base logging RINEX data and Sattelite Navigation data to the corresponding places. Specifically,
- Rover: `.O` file from your GNSS rover logging RINEX data
- Base: `.O` file from CORS logging RINEX data
- Navigation: `.P` file from your GNSS rover logging RINEX data
In this post, you will see the files are with extended name `.26O` and `.26P`, the two digits 26 means year 2026, so you may see `.27O` and `.27P` if you are collecting data in 2027.
 
**Step 7**. Click ⌈Process⌋ to run PPK. Once it is done, you will see the trajectory of your GNSS receiver in centermiter level.

**Step 8**. Import your survey CSV file, and generate corrected CSV using PPK. The generated files will be saved in the Rover logging RINEX data folder by default, or you can also find it by clikcing ⌈Show result files⌋



## Supplementary
### Understanding UTC Time (Very Important)

GNSS systems use UTC time, not your local time.

However, when you perform the survey in the field, you usually record the local time. This can easily cause confusion when downloading CORS data.

For example, if you are working in New York (Eastern Time):

| Local Time | UTC Time |
|----|----|
| 08:00 | 12:00 |
| 09:00 | 13:00 |
| 10:00 | 14:00 |
| 11:00 | 15:00 |

This means:

Local time + 4 hours = UTC (When daylight saving time is active.)

If you perform a survey at 11:00 AM EST, the corresponding UTC time is 15:00 UTC. This is the time you must use when downloading CORS data.

### Understanding CORS File Time Codes

CORS hourly files use letters to represent hours.
| Letter | Hour (UTC) |
|----|----|
|a| 00|
|b| 01|
|c| 02|
|d| 03|
|e| 04|
|f| 05|
|g| 06|
|h| 07|
|i| 08|
|j| 09|
|k| 10|
|l| 11|
|m| 12|
|n| 13|
|o| 14|
|p| 15|
|q| 16|
|r| 17|
|s| 18|
|t| 19|
|u| 20|
|v| 21|
|w| 22|
|x| 23|

Example file: `nywl075p.26o`

Meaning:

| Part | Meaning|
|----|----|
|nywl| station ID|
|075| day of year|
|p| hour 15 UTC|
|26| year 2026|
|o| observation file|

So this file contains data for 15:00–16:00 UTC on the 75th day in 2026 at NYWL station.

### How to Choose the Correct CORS Data

To run PPK successfully, the base station data must overlap with your survey time.

Example:

Survey time:
- Local time: 11:05 – 11:20
- UTC time: 15:05 – 15:20

You must download:
- 15:00–16:00 UTC data 
Which corresponds to letter `p` -> nywl075p.26o

Recommended practice: to avoid time mismatch problems, download two adjacent hours of data.
- `nywl075o.26o`
- `nywl075p.26o`
This ensures the base station data fully covers your survey period.

### Common Reasons Why PPK Fails

If PPK processing fails, the problem is usually caused by one of the following issues.

1. Base station too far away
If the base station is too far from the survey location, atmospheric errors become too large.

2. Incorrect survey time
- using local time instead of UTC
- downloading the wrong hour file
- base data does not overlap with rover data

If the base data does not cover the rover time, PPK cannot compute corrections.

3. Wrong file types
- Rover observation: `.yyO`
- Base observation: `.yyO`
- Navigation: `.yyP`
Using the wrong file type will cause processing errors.

4. Downloaded CORS data doesn't contain valid data
这可能是因为你输入了错误的代码，导致404 not found之类的，检查你的.O数据是否包含和来自rover的.o数据相同格式的数据

5. Survey points recorded too quickly
If the averaging duration is too short, there may not be enough satellite observations to compute corrections.

### Final Advice
When performing PPK surveys:
- Choose the closest CORS station
- Always check UTC time
- Download correct hourly files
- Ensure survey duration ≥ base sampling interval
- Verify the correct RINEX file types
Following these steps will ensure reliable centimeter-level positioning.
